# 台北市運動中心場地預約查詢 App 需求文件

**專案代號：** tpsvb (Taipei Public Sports Venue Booking)
**平台：** Android (Flutter)
**文件版本：** v1.3
**建立日期：** 2026-02-27
**最後更新：** 2026-02-28

---

## 1. 背景與問題描述

### 1.1 現況問題

台北市公立運動中心統一預約系統（https://booking-tpsc.sporetrofit.com/）雖提供線上預約功能，但使用體驗極差：

- 每次僅能查看**單一運動中心、單一運動類型、單一日期、單一時段**的場地狀態
- 使用者必須重複點擊才能跨場館、跨日期查詢是否有空位
- 無法一次總覽多個場館、多個時段的可預約狀況
- 對於想在空閒時間出門運動的使用者，找到合適時段需要花費大量時間

### 1.2 目標使用者

- 台北市民，習慣於台北各公立運動中心從事羽球、籃球、桌球、壁球、撞球、高爾夫球等運動
- 希望快速找出「近期哪個時段、哪個場館有空位」的使用者
- 有明確運動偏好（特定運動項目），希望比較多家場館的使用者

---

## 2. 專案目標

開發一款 Android App，透過呼叫台北市運動中心預約系統的後端 API，**一次撈取多個場館、多個日期的時段可用狀態**，以清晰的總覽畫面呈現，讓使用者能快速找到可預約的場地。

## 3. 功能需求

### 3.1 核心功能

#### F-01｜啟動首頁（Splash / Landing Page）

- App 啟動後顯示專屬首頁，內容包含：
  - 系統全名：**臺北市公立運動場地預約查詢系統**
  - 副標題或說明文字（例：快速瀏覽各運動中心場地預約狀況）
  - **開始查詢**按鈕
- 點擊畫面任意位置**或**點擊「開始查詢」按鈕，皆可進入查詢主畫面
- 首頁不做自動跳轉計時器，讓使用者主動決定何時進入

#### F-02｜場館總覽（查詢主畫面）

- 以**日期 × 時段 × 場館**的矩陣式表格或列表方式，同時顯示多筆場地的可用狀態
- 狀態以顏色區分：
  - **可預約（綠色）**
  - **已額滿（紅色）**
  - **未開放預約（灰色）**
  - **載入中（骨架屏動畫）**
- **逐步顯示**：哪個場館的資料先回來就先顯示，不等全部完成，格子從「載入中」逐一更新為實際狀態
- 支援橫向（場館/運動類型）與縱向（日期/時段）的交叉查詢
- 畫面固定顯示**場地預約須知入口**（詳見 F-08）

#### F-03｜資料來源：API 串接

- 透過呼叫 `booking-tpsc.sporetrofit.com` 後端 API 取得場地預約狀態資料
- 支援同時查詢多個場館（支援以下 11 個行政區的運動中心）：
  - 中正、內湖、北投、大安、大同、士林、松山、萬華、文山、信義、中山
- **每次查詢固定為單一球類**，涵蓋範圍：使用者選定運動類型 + 未來 7～14 天 + 全天各時段
- 資料強調即時性，**不做背景預取**，每次查詢都向伺服器取得最新狀態（預約名額可能隨時因他人取消而釋出）

#### F-04｜篩選與搜尋

- **運動類型篩選**：羽球（Badminton）、籃球（Basketball）、撞球（Billiard）、高爾夫球（Golf）、壁球（Squash）、桌球（TableTennis）
- **場館選擇**：可勾選要查詢的特定場館（預設：全選）
- **日期範圍選擇**：可指定查詢的日期區間（預設：今天起 7 天）
- **時段篩選**：可選擇早上 / 下午 / 晚上區間

#### F-05｜快速跳轉預約

- 點擊某一「可預約」的時段格，直接以外部瀏覽器開啟對應的預約頁面（深度連結至官方預約網站）

#### F-06｜資料快取

- 查詢結果**短暫**記憶體快取（快取時效：5 分鐘），防止使用者誤觸重新查詢時重複打 API
- 快取時效刻意設短，確保資料接近即時（預約名額隨時可能因取消而釋出）
- 使用者可手動下拉強制更新，清除快取重新取得最新資料

#### F-07｜我的偏好設定

- 儲存常用的運動類型與場館偏好，下次開啟自動套用
- 設定儲存於本機（SharedPreferences / local storage）

#### F-08｜場地預約須知

- 查詢主畫面提供固定入口（例：右上角「須知」按鈕或底部連結），點擊後以 App 內建 WebView 開啟官方預約須知頁面：
  `https://booking-tpsc.sporetrofit.com/Home/BookingInformation#1`
- 以 WebView 呈現（而非跳出外部瀏覽器），讓使用者看完後可直接返回 App
- 須知內容由官方網站提供，包含各運動中心的預約開放時間、付款期限、取消規則等重要資訊

## 4. 非功能需求

### 4.1 效能

- 畫面首次出現內容（第一個場館資料）：≤ 2 秒
- 全部場館資料載入完成：≤ 10 秒（11 場館並發，視網路狀況）
- 有快取時畫面立即呈現（≤ 0.5 秒），快取時效 5 分鐘
- 並發（concurrent）API 請求，分批控制同時在途請求數（建議最大並發數：5）

### 4.2 可用性

- 支援 Android 8.0（API Level 26）以上
- 支援手機直向使用（主要）；平板橫向可用（次要）
- 空狀態提示（無可預約場地時顯示友善訊息）
- 錯誤狀態提示（網路錯誤、API 失敗時顯示錯誤訊息與重試按鈕）

### 4.3 安全性與合規

- 本 App 僅**讀取**公開可查詢的預約狀態資料，不執行任何預約或付款操作
- 不收集、不儲存使用者個人資料
- 請求行為須符合對方網站的 robots.txt 及使用規範；請求頻率應合理，避免對伺服器造成過大負荷

## 9. API 規格（已確認）

### 9.1 API Endpoint 一覽

所有 API 均透過 POST 請求呼叫，Base URL：`https://booking-tpsc.sporetrofit.com`

#### API-01｜查詢特定場館、球類在日期區間內開放預約的日期

```
POST /Location/findAllowBookingCalendars
Content-Type: application/x-www-form-urlencoded

Form Data:
  LID=<場地代號>        // 例：ZSSC
  categoryId=<球類>     // 例：Badminton
  start=<yyyy-mm-dd>    // 例：2026-02-28
  end=<yyyy-mm-dd>      // 例：2026-03-07
```

**用途：** 取得哪些日期有開放預約，可用來過濾無需查詢的日期，減少後續請求數量。

#### API-02｜查詢特定場館、球類、特定日期的場地與時段清單

```
POST /Location/findAllowBookingList?LID=<場地代號>&categoryId=<球類>&useDate=<yyyy-mm-dd>
Content-Type: application/x-www-form-urlencoded

Form Data:
  rows=200
  page=1
  sord=asc
```

**用途：** 取得指定日期下所有可預約的場地名稱、時段與費用（最多 200 筆）。

### 9.4 請求量化分析

#### 最壞情況（不優化）

```
11 場館 × 7 天 = 77 次 API-02 請求
+ 11 場館 × 1 次 API-01 = 11 次
─────────────────────────────
總計最多 88 次請求
```

#### 優化後（兩段式查詢）

```
第一階段：11 次 API-01（並發）
  → 取得各場館在 7 天內有哪些日期有開放
  → 假設平均每館有 5 個有效日期（最多 7 天）

第二階段：11 場館 × 5 有效日期 = ~55 次 API-02（並發）
  → 跳過未開放日期，避免無效請求

實際總請求數：約 55～77 次（視各場館開放程度）
```

### 9.5 體驗優化策略

由於即使優化後仍需約 55～77 次請求，需透過以下設計避免使用者感受到長時間等待：

#### 策略一：逐步顯示（Progressive Loading）

- 每個場館的格子初始狀態為「骨架屏（Skeleton）」
- 哪個場館的 API 回傳結果，該場館的格子立即更新，不等其他場館
- 技術實作：每個場館對應獨立的 `AsyncNotifier`（Riverpod），各自管理自己的載入狀態

```
畫面初始狀態：
┌──────┬────────┬────────┬────────┐
│      │ 2/28   │  3/1   │  3/2   │
├──────┼────────┼────────┼────────┤
│ 中山 │ ░░░░░░ │ ░░░░░░ │ ░░░░░░ │  ← 骨架屏
├──────┼────────┼────────┼────────┤
│ 中正 │ ░░░░░░ │ ░░░░░░ │ ░░░░░░ │  ← 骨架屏
├──────┼────────┼────────┼────────┤
│ 內湖 │ ░░░░░░ │ ░░░░░░ │ ░░░░░░ │  ← 骨架屏
└──────┴────────┴────────┴────────┘

中山資料最先回來後：
┌──────┬────────┬────────┬────────┐
│ 中山 │  ● 可  │  ✗ 滿  │  ● 可  │  ← 立即更新
├──────┼────────┼────────┼────────┤
│ 中正 │ ░░░░░░ │ ░░░░░░ │ ░░░░░░ │  ← 仍載入中
```

#### 策略二：分批並發（Batched Concurrent Requests）

避免一次送出 77 個請求導致手機或伺服器過載：

```
第一階段（API-01）：
  同時送出 11 個請求 → 全部完成後進入第二階段

第二階段（API-02）：
  以每批 5 個請求並發，批次間不強制等待
  （使用 stream 或 isolate pool 控制最大並發數）
  哪一批完成就立即更新對應場館的畫面格子
```

#### 策略三：優先顯示「可預約」結果

- 若某場館某日期的 API-01 回傳「無開放日期」，立即將該場館所有格子標記為「未開放（灰色）」，使用者可快速跳過
- 已確認可預約的時段優先顯示，讓使用者不需等待所有資料才能行動

#### 策略四：進度指示

- 畫面上方顯示進度條或文字（例：「載入中 7 / 11 場館」），讓使用者知道還在進行中
- 全部完成後進度條消失，顯示「最後更新：XX:XX」時間戳

## 11. 風險與限制

| 風險 | 說明 | 因應策略 |
|------|------|---------|
| API 結構可能變動 | 官方可能隨時改版 endpoint 或參數 | 架構中抽離 API 解析層，方便快速更新 |
| 請求頻率限制 | 最多 88 次請求，可能觸發伺服器限流 | 兩段式查詢 + 分批並發（最大 5 個）控制請求速率 |
| API 可能需要登入 | 尚未確認 guest 模式是否完全可用 | 實際測試後確認；若需登入則提示使用者 |
| 各場館球類不同 | 非每個場館都有全部 6 種球類 | 以 API-01 回傳結果動態判斷，無支援時標記灰色 |
| 網路環境差時載入緩慢 | 使用者在弱網環境下可能等很久 | 逐步顯示設計讓使用者能看到部分結果；超時後顯示重試 |

## 12. 範疇外（Out of Scope）

以下功能**不在**本版本開發範圍內：

- 直接在 App 內完成預約（需整合登入與付款，法律與安全風險高）
- iOS 版本
- 推播通知（有空位時通知）—— 可列為 v2.0 功能
- 用戶帳號系統
- 游泳池即時人數查詢（雖官網有此功能，但屬附加功能）
